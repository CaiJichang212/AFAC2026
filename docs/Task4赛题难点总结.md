赛题要求构建一个金融长文本 Agent（智能体），在推理问答阶段只能调用 Qwen 系列模型 API，不微调模型，通过检索、记忆压缩、上下文管理和证据推理，尽量少用 Token（词元）回答金融长文档选择题/判断题。

核心任务：
从保险条款、监管法规、金融合同、财报、研报五类长文档中，找到与题目相关的证据，判断 A/B/C/D 选项真假，输出规范答案，并统计 Token 消耗。

主要难点：

1. 长文档太长，不能全文输入模型，必须做高质量切分、检索和压缩。
2. 金融文本细节多，期限、比例、金额、例外条件、条款编号都可能决定答案。

8. 不能用非 Qwen 模型参与正式检索、排序、推理或纠错，限制了常规 RAG（检索增强生成）方案。

这是一个考察“金融长文档精准检索 + 证据级推理 + 动态上下文压缩 + Token 成本控制”的 Agent 系统赛题，核心胜负点在于找准证据、逐项判断选项、稳定输出规范答案。

## 一句话理解赛题

这是一个“金融长文本 Agent（智能体）问答系统”赛题：在不能微调基座模型、推理阶段只能调用 Qwen 系列 API 的前提下，设计一套能对金融长文档进行解析、检索、压缩、证据聚合、推理和答案规范化的系统，用尽量少的 Token（词元）准确回答单选、多选、判断题。赛题核心不是单纯问答，而是“长文档信息管理 + 金融证据推理 + 成本控制”的综合工程问题。

## 任务本质

赛题输入是多类金融长文档和问题集，输出是 `answer.csv`，每题只提交答案字母，同时提交本次评测的 Token 消耗统计。题型包括单选题、多选题、判断题；多选必须按字母顺序输出，漏选、错选、多选都算错，没有部分分。

数据覆盖五类金融文本：保险条款、监管法规、金融合同、财务报表、行业研报，共 200 道题，其中 A 榜 100 道、B 榜 100 道。A 榜提供相对明确的 `doc_ids`，更适合调试基础阅读能力；B 榜不直接给 `doc_ids`，需要系统先做候选文档检索，再做阅读理解和答案判断，更接近真实盲测场景。

评分以准确率为主，Token 效率为辅。最终分数公式是：

`FinalScore = 100 * Accuracy * (0.7 + 0.3 * TokenScore)`

其中 `TokenBudget = 5,000,000`，Token 效率最多影响 30% 权重。因此，系统首先要保证准确率，然后再压缩 Token；盲目省 Token 导致错题，不划算。

## 关键限制

最重要的限制是：预处理阶段可以使用 OCR（光学字符识别）、版面分析、表格恢复、PDF 转结构化文本等工具，甚至可以使用非 Qwen 模型做文档解析；但正式推理问答阶段，包括文档定位、段落检索、记忆压缩、证据判断、答案生成，必须严格使用 Qwen 系列模型。非 Qwen 模型不能生成向量、排序结果、召回结果、rerank（重排序）、答案投票或纠错，也不能生成语义摘要、FAQ、结论型知识库参与正式答题。

这意味着常规“用强 embedding（嵌入）模型 + reranker（重排序器）+ LLM 回答”的 RAG（检索增强生成）方案在这里会受到明显限制。更可行的是构建规则检索、关键词检索、结构化索引、领域字段索引，再用 Qwen 完成必要的语义判断和推理。

## 赛题主要难点

第一，文档长且结构复杂，不能直接全文塞给模型。赛题文件明确指出，金融长文本往往超过模型上下文窗口的经济使用区间，全文输入会导致 Token 成本巨大；简单截断或朴素摘要又会丢失关键事实、业务逻辑和约束条件。基线结果也说明，直接输入长文档的 Qwen-plus 方案总准确率只有 15.0%，Token 消耗达到 7,512,231，效果和成本都不理想。

第二，检索必须非常精准。金融问题经常依赖具体条款、页码、章节、表格、指标名、年份、比例、例外条件。普通关键词匹配容易漏掉同义表述，例如“身故保险金”“死亡给付”“保险责任”；也容易被相似条款干扰，例如“普通决议”和“特别决议”、“现金价值”和“账户价值”、“营业收入”和“归母净利润”。所以检索要支持题干检索、选项级检索、条款编号检索、财务指标检索、领域术语扩展和多轮补检索。

第三，B 榜的文档定位更难。A 榜有 `doc_ids`，可以直接在相关文档内找证据；B 榜没有 `doc_ids`，系统要先判断题目属于哪个领域、涉及哪些实体、哪些年份、哪些产品或法规，再从所有候选文档中定位正确文档。错误的文档召回会直接导致后续推理失败。

第四，跨文档、跨章节证据聚合是高频需求。财报题可能要求比较两年年报；保险题可能要求比较多个产品的赔付规则；监管题可能同时涉及多个法规文件的适用条件和优先级；研报题可能需要对比不同公司或行业指标。单段命中不够，系统需要把多个证据片段合并成稳定的中间事实。

第五，数值计算和条件推理容易出错。题目会涉及比例、金额、期限、现金流、责任范围、资产负债率、净资产占比、研发投入占比、保险金计算公式等。难点不只是计算本身，而是先识别正确的计算口径，再结合触发条件判断选项。例如“超过 70%”“不低于”“未达到”“任一条件触发”“同时满足”等条件词非常关键。

第六，动态记忆压缩本身很难。系统需要在多轮检索和阅读过程中保留关键事实，同时压掉无关上下文。压缩过度会丢证据，压缩不足会浪费 Token。理想的“记忆”不应是普通摘要，而应是面向题目的结构化证据状态，例如：实体、指标、年份、数值、条款编号、适用条件、例外条件、支持/反驳哪个选项。

第七，答案格式容错空间很小。最终只要答案字母非法、顺序不规范，或者多选题多一个/少一个选项，都会错误。赛题还要求 Token 统计覆盖检索摘要、上下文压缩、证据判断、答案生成和自检等所有模型调用。因此后处理必须强约束：只输出合法字母，单选/判断取唯一答案，多选去重排序，并记录 Token。

第八，领域差异明显，不能一套 Prompt（提示词）吃到底。保险条款重在责任触发、给付规则、现金价值、退保金额；监管法规重在条文适用、时间期限、处罚、决议方式、义务主体；金融合同重在债券条款、评级、发行信息、权利义务；财报重在指标抽取、同比比较、现金流和分红；研报重在观点核验、公司比较、行业趋势。不同领域需要不同的切分、索引、证据模板和推理 Prompt。

## 难点优先级判断

最高优先级难点是“证据召回正确性”。如果找不到正确文档和正确片段，后面的模型再强也只能猜。

第二优先级是“选项级证据判断”。多选题没有部分分，必须逐项判断每个选项是否被证据支持、是否存在偷换概念、是否把普通决议说成特别决议、是否把某个条件说成充分条件。

第三优先级是“Token 成本控制”。评分中 Token 效率最多影响 30%，但准确率仍是主体。因此策略上应先建立高召回、可追溯的解题链路，再优化重复检索、全文输入和冗余模型调用。

第四优先级是“可复现工程”。B 榜前 15 名需要提交完整代码、`evidence.json`、处理后数据、Agent 系统、脚本、日志和复现文档。也就是说方案不能只是 notebook 临时跑通，而要能一键复现。

## 总结

这个赛题的核心挑战不是让大模型“读懂金融文档”这么简单，而是在强约束下构建一个可靠的金融长文本问答流水线：文档解析要保留结构，检索要精准，证据要可追溯，压缩要不丢关键事实，推理要逐选项严谨，输出要完全规范，同时还要控制 Token 成本。

从竞赛策略看，最值得投入的方向是：高质量 PDF 解析与结构保留、领域化混合检索、选项级证据聚合、结构化动态记忆、答案合法化后处理，以及按保险/法规/合同/财报/研报分别设计推理模板。
