# 金融Agent问答系统需求文档

版本：v1.0  
日期：2026-06-13  
适用赛题：金融长文本 Agent 的动态记忆压缩与高效问答挑战  
交付物：本需求文档、赛题数据分析报告、技术调研与调用报告、技术架构文档

## 1. 项目背景

金融文本问答面向保险条款、监管法规、债券募集说明书、上市公司年报、行业研报等长文档。这类文档通常包含期限、比例、例外条件、权利义务、跨章节引用、表格指标和复杂数值口径。系统不能依赖通用知识或简单关键词匹配，而需要在多份长文档中定位证据、理解约束、完成计算，并输出严格可评测的答案字母。

赛题限制要求：正式问答推理阶段只能调用 Qwen 系列模型 API，不能修改基座模型参数；文档解析、OCR（光学字符识别）、版面分析、表格恢复等预处理阶段可以使用非 Qwen 工具，但不能把非 Qwen 模型生成的向量、语义排序、摘要、FAQ 或结论带入正式答题流程。系统必须生成 `answer.csv`，同时统计每道题和全量任务的输入 Token、输出 Token、总 Token。

## 2. 项目目标

建设一个“金融Agent问答系统”，在给定题目和金融长文档材料的条件下，自动完成文档定位、证据检索、动态记忆压缩、逐选项判断、答案规范化和 Token 统计。系统目标是优先提升准确率，同时在 5,000,000 Token 预算下尽量降低总 Token 消耗。

核心目标如下。

1. 准确回答单选题、多选题、判断题，并满足答案格式约束。
2. 对每道题输出可追溯证据，便于复盘、调试与代码审核。
3. 在 A 榜利用 `doc_ids` 构建高置信证据链；在 B 榜缺失 `doc_ids` 时完成候选文档检索。
4. 避免直接全文输入，采用多级检索和动态压缩控制 Token 成本。
5. 严格遵守模型使用限制，确保正式推理流程只使用 Qwen 系列模型。

## 3. 范围定义

### 3.1 范围内

- PDF、HTML、TXT 等原始文档解析为 Markdown/JSONL（JSON Lines）结构化文本。
- 章节、页码、表格、条款编号、标题、页眉页脚、脚注等元数据保留。
- 构建文档清单、字段索引、关键词索引、条款索引、数值索引和可选的 Qwen 系列向量索引。
- 题目解析：识别领域、题型、实体、年份、金额、比例、期限、指标、选项声明。
- A 榜按 `doc_ids` 限定文档范围；B 榜先做候选文档召回。
- 证据检索、证据压缩、逐选项判断、答案汇总、答案合法化。
- 生成 `answer.csv`、`evidence.json`、`logs/`、`processed_data/` 和复现脚本。

### 3.2 范围外

- 不训练、不微调、不蒸馏 Qwen 基座模型。
- 不使用非 Qwen 模型参与正式检索、排序、答案生成、投票或纠错。
- 不把预处理阶段生成的语义摘要作为正式答题证据。
- 不使用互联网检索来回答赛题问题；赛题问题只能基于给定文档。

## 5. 业务难点

第一，文档长且结构复杂。金融合同与年报的 A 题引用文档平均超过 280 页，合同领域部分文档超过 500 页，直接全文输入会导致成本高、噪声大、证据稀释。

第二，题目多数为选项级事实核验。题干通常给出比较任务，选项分别声明具体指标、条款或结论。系统若只生成最终答案，容易受干扰项影响；必须把选项拆为可验证声明。

第三，多文档比较频繁。财报题需要跨年份、跨公司比较；保险题常需要 2 到 4 款产品联合计算；合同题需要对发行要素、评级、中介机构、违约条款进行横向比对。

第四，数值口径复杂。金额单位可能是元、万元、亿元；年报口径可能是归母净利润、扣非净利润、经营现金流、研发投入占比；保险题涉及已交保费、现金价值、账户价值、年龄系数、领取前后差异。

第五，监管法规存在条文层级与适用时点。法规题要求定位条、款、项，区分“应当”“可以”“不得”，并判断施行日期、报告时限、审议程序、特别决议等条件。

第六，PDF 解析质量直接影响上限。表格、目录、双栏、页眉页脚、扫描页、附件、跨页表格会导致检索片段缺漏或顺序错乱。

第七，评测强约束答案格式。单选与判断只允许一个字母；多选需要去重、排序、无分隔符。输出非法字符会直接损失分数。

第八，Token 统计覆盖所有模型调用。即使调用用于检索摘要、证据判断、答案自检，也必须计入 `summary.total_tokens`。缓存可降低费用或延迟，但提交统计应以 API usage 为准，而不是按缓存折扣改写。

## 6. 功能需求

### FR-01 文档解析与结构化

系统应将 PDF/HTML/TXT 解析为统一的 `DocumentBlock` 结构。每个块应包含 `doc_id`、`domain`、`source_path`、`page_start`、`page_end`、`section_path`、`block_type`、`text`、`table_html`、`bbox`、`hash`。PDF 解析应尽量保留阅读顺序、表格标题、表格单元格、条款编号和页码。

验收标准：随机抽样每个领域至少 5 份文档，核心标题、页码、表格字段、条款编号可检索；无严重乱码；跨页表格能被关联或标记。

### FR-02 文档清单与 doc_id 映射

系统应自动生成 `doc_manifest.json`，将题目 `doc_ids` 映射到原始路径、解析后路径、标题、领域、页数、字符数和文件哈希。对于监管领域，应处理 `csrc_xxxx_attn`、`strict_v3_*`、HTML 文件和附件文件的命名差异。

验收标准：A 榜所有 `doc_ids` 能唯一映射；B 榜可通过标题、正文、文件名生成候选文档。

### FR-03 多粒度切分

系统应生成多粒度 Chunk：段落块、条款块、表格块、页面窗口块、父子块。默认段落块控制在 600 到 900 个中文字符，重叠 80 到 150 字符；表格块按表名、列名、行组和页码组织；条款块按“第X条/第X章/（一）”层级切分。

验收标准：同一证据附近可通过 `parent_id` 扩展上下文；切分不切断关键公式、金额单位、条款编号。

### FR-04 检索系统

系统应实现正式合规的检索流程。基础模式使用 BM25（最佳匹配25）、短语匹配、正则匹配、字段过滤、条款编号匹配、数值匹配和 RRF（倒数排序融合）。如赛题方确认 `text-embedding-v4` 属于允许的 Qwen 系列，可启用 Qwen Embedding（Qwen 嵌入）作为可选语义召回；否则关闭向量检索。

验收标准：A 榜在给定 `doc_ids` 内 Top-K 证据召回率可人工抽样验证；B 榜候选文档 Top-5 命中率成为核心离线指标。

### FR-05 题目解析

系统应解析 `qid`、`domain`、`answer_format`、`doc_ids`、题干、选项。应识别关键实体、年份、公司、产品、法规、金额、比例、期限、动作词、否定词。解析优先用规则；必要时调用 Qwen 生成结构化 JSON。

验收标准：题目解析结果字段完整；多选题每个选项形成独立声明；数值单位被标准化。

### FR-06 证据检索与扩展

系统应针对题干与每个选项构造多路查询，包括题干查询、选项查询、实体+指标查询、条款编号查询、数值查询。检索后对命中块进行相邻页扩展、同表扩展、父节点扩展和跨文档聚合。

验收标准：每个选项至少返回 2 到 5 条候选证据；若证据不足，触发二次查询。

### FR-07 动态记忆压缩

系统应维护 `EvidenceMemory`（证据记忆），由原文证据、结构化事实和冲突状态组成。压缩不应替代原始证据，而应服务于最终 Prompt（提示词）预算控制。压缩输出必须包含来源、数值、条件、适用范围和支持/反驳选项。

验收标准：最终判断 Prompt 中保留可追溯原文片段；压缩后的事实能追溯到 `evidence_id`。

### FR-08 Qwen 推理与逐选项判断

系统应调用 Qwen 系列模型逐选项判断。输出结构应包含 `option`、`verdict`、`confidence`、`evidence_ids`、`calculation`、`short_reason`。最终答案由确定为 `true` 的选项组合而成。

验收标准：模型输出为合法 JSON；若 JSON 解析失败，自动修复或重试一次；不得输出自由文本作为最终答案来源。

### FR-09 数值计算与单位规范化

系统应在 Python 中进行确定性计算，避免让模型口算复杂比例。支持单位换算、同比/增减判断、现金价值/账户价值比较、比例阈值判断、日期差计算。模型负责解释条件与选择公式，计算由程序执行。

验收标准：所有计算过程写入 `evidence.json`；数值输出保留原始单位与标准单位。

### FR-10 答案规范化与提交文件生成

系统应根据题型生成合法答案：`mcq` 和 `tf` 输出单个大写字母，`multi` 输出去重、升序、无分隔符的大写字母串。系统应生成 `answer.csv`，首行 `summary` 汇总全量 Token。

验收标准：`answer.csv` 列为 `qid,answer,prompt_tokens,completion_tokens,total_tokens`；无非法字符；每道题有 Token 统计。

### FR-11 证据可追溯与日志

系统应生成 `evidence.json`，保存候选文档、检索分数、证据片段、模型判断、计算过程、最终答案和 Token usage。日志应能复现每题的关键中间结果。

验收标准：任意一道题可定位到原始文件、页码、chunk_id、证据文本、模型调用 ID 和答案生成过程。

### FR-12 合规开关

系统应提供 `compliance_profile` 参数：`strict_competition`、`qwen_embedding_allowed`、`debug_local`。正式提交必须使用 `strict_competition`，禁用非 Qwen 语义向量、非 Qwen rerank、非 Qwen 答案纠错。

验收标准：运行日志记录所有模型调用的 provider 和 model；正式模式下若检测到非 Qwen 推理模型，直接中止。

## 7. 非功能需求

准确性优先于成本。在相同准确率下，降低 Token 消耗。系统应优先减少无效全文输入、重复模型调用和宽泛证据片段。

可复现性要求一键运行：`python script/run_all.py --config configs/strict_competition.yaml`，输出 `answer.csv` 与 `evidence.json`。

可维护性要求模块边界清晰，检索、压缩、推理、后处理均可独立替换。

鲁棒性要求处理 PDF 解析失败、JSON 输出失败、API 超时、速率限制、缺证据、冲突证据、非法答案等异常。

安全性要求 API Key 不写入日志或提交包；原始数据和中间文件路径可配置。

性能要求 A 榜 100 题离线运行在可接受时间内完成；正式 B 榜应支持断点续跑、并发限流和失败重试。

## 8. 验收指标

核心指标是 Accuracy（准确率）。内部还应增加以下离线指标。

- DocRecall@5：B 榜候选文档 Top-5 命中率。
- EvidenceRecall@K：人工抽样证据召回率。
- Option-F1：逐选项判断精度、召回率和 F1。
- InvalidAnswerRate：非法答案率，应为 0。
- AvgPromptTokensPerQuestion：平均输入 Token。
- TokenPerCorrectAnswer：每个正确答案消耗的 Token。
- SelfCheckTriggerRate：二次自检触发比例。

## 9. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| PDF 表格解析错误 | 财报、合同、研报数值题错误 | PyMuPDF + pdfplumber + Camelot/MinerU 多解析器交叉校验；表格保留页码与截图索引 |
| 多选题漏选或多选 | 直接判错 | 逐选项二分类，答案由程序合成；冲突选项触发自检 |
| B 榜无 doc_ids | 文档定位失败 | 构建标题索引、正文 BM25、条款/实体/数值索引；按领域路由 |
| 非 Qwen 语义结果误用 | 违规风险 | 正式 profile 禁止非 Qwen 向量、非 Qwen rerank、非 Qwen 语义摘要 |
| Token 超预算 | 综合分下降 | 分层检索、证据裁剪、只对不确定题自检、复用候选证据 |
| Qwen JSON 输出不稳定 | 后处理失败 | 使用结构化输出或 JSON schema，失败后一次修复重试 |

## 11. 结论

本项目的需求核心不是“长上下文直接问答”，而是“可审计证据驱动的低成本选项验证”。系统必须以文档解析质量、混合检索、选项级判断、动态记忆压缩和答案规范化为主线建设。正式推理链路应尽量保守，确保只使用 Qwen 系列模型，同时保留足够日志证明合规与可复现。
