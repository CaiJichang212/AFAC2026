# 金融Agent问答系统赛题数据分析报告

版本：v1.0  
日期：2026-06-13  
数据包：`public_dataset_a.zip`  
说明：本报告基于用户提供的公开 A 榜数据包与赛题说明文件进行解析统计。统计中的“文本字符数”来自本地 PDF/HTML/TXT 抽取结果，用于估计检索和上下文压缩难度，不代表官方 Token 统计。

## 1. 数据包结构

数据包顶层目录为 `public_dataset_upload/`，包含两类内容：

```text
public_dataset_upload/
├── questions/group_a/
│   ├── financial_contracts_questions.json
│   ├── financial_reports_questions.json
│   ├── insurance_questions.json
│   ├── regulatory_questions.json
│   └── research_questions.json
└── raw/
    ├── financial_contracts/
    ├── financial_reports/
    ├── insurance/
    ├── regulatory/
    └── research/
```

公开包只包含 A 榜题目，合计 100 道。每个领域 20 道，且所有 A 榜题均提供 `doc_ids`。正式 B 榜不会直接给出 `doc_ids`，因此 A 榜应主要用于验证文档解析、证据检索、逐选项判断和答案规范化；B 榜需要额外评估候选文档定位能力。

## 2. 题目字段结构

题目 JSON 的主要字段如下。

| 字段 | 含义 | 对系统的影响 |
|---|---|---|
| `qid` | 题目唯一编号 | 用于日志、答案文件和证据文件主键 |
| `domain` | 领域 | 用于文档路由、领域 Prompt、指标规则 |
| `split` | A/B 组 | A 组有 doc_ids；B 组需文档检索 |
| `question` | 题干 | 提取任务意图、实体、年份、指标、条件 |
| `options` | A/B/C/D 选项 | 应拆成独立待验证声明 |
| `answer_format` | `mcq`/`multi`/`tf` | 决定答案后处理规则 |
| `type` | 题目能力标签 | 可用于 Prompt 模板选择 |
| `doc_ids` | 关联文档编号 | A 榜直接限定候选文档；B 榜缺失 |

## 7. 各领域数据特征与难点

### 7.1 insurance：保险条款

保险文档体量小，但题目多为计算、排序和条件触发。常见证据包括身故保险金、现金价值、账户价值、已交保费、领取日前后、年龄系数、退保规则。系统需要抽取公式，并在 Python 中执行确定性计算。保险题的高风险点是公式适用条件，而不是公式本身。

建议：构建产品级结构表：`product_name`、`benefit_type`、`trigger_condition`、`formula`、`min/max rule`、`source_clause`。最终判断时让 Qwen 选择公式和条件，计算由程序执行。

### 7.2 regulatory：监管法规

监管原始目录包含 HTML、PDF 附件和 TXT。A题引用文档相对集中，但 B 榜候选空间大。难点在于条款编号、施行日期、义务主体、报告时限、监管措施、特别决议、禁止性表述。选项常包含“必须”“可以”“不得”“普通决议”“特别决议”等强逻辑词。

建议：建立法规条文层级索引，字段包括 `law_title`、`article_no`、`clause_no`、`effective_date`、`obligation_subject`、`deadline`、`must/can/prohibit`。检索时优先条文编号与关键词短语，避免语义近似误召回。

### 7.3 financial_contracts：金融合同/债券募集说明书

合同文档页数最多，证据分散在发行摘要、风险提示、评级、募集资金用途、受托管理、违约责任、偿债保障等章节。问题常要求两份募集说明书对比，涉及发行主体、金额上限、信用评级、中介机构、时间节点、条款差异。

建议：预处理阶段生成章节目录；重点抽取“发行概况”“评级情况”“受托管理人”“募集资金用途”“违约责任”“债券持有人会议”等章节。检索时以章节标题加选项实体双路查询。

### 7.4 financial_reports：上市公司年报

财报文档平均超过 280 页，核心证据大量存在于表格。题目多为连续两年或两公司指标比较。难点包括单位换算、同比方向、口径一致性、分红比例、研发投入占比、现金流、归母净利润等。

建议：将关键财务表转换为结构化表格，列包含 `company`、`year`、`metric`、`value`、`unit`、`page`、`source_table`。对常见指标做别名词典，如“归属于上市公司股东的净利润/归母净利润”。

### 7.5 research：行业研报

研报文字量中等，但信息多在图表、预测数据、行业对比和研究结论。选项常引用具体年份、市场规模、同比增速、行业趋势和公司指标。PDF 解析如果丢失图表标题或表格注释，答案容易错。

建议：保留图表标题、图表周边段落、数据来源、单位和注释。对研报采用图表块 + 周边段落块联合检索。

## 8. 数据质量风险

| 风险 | 体现 | 影响 |
|---|---|---|
| 文件命名与题目 doc_id 不完全语义化 | 如 `text01`、`1.pdf`、`pack2_text01` | B榜文档定位必须依赖正文与标题索引 |
| PDF 扩展名大小写混合 | `.PDF` 与 `.pdf` | 路径匹配应大小写不敏感 |
| 监管文件多格式并存 | HTML、PDF 附件、TXT | 需要统一 doc_id 规则和正文清洗 |
| 表格跨页与单位隐藏 | 财报、合同、研报 | 数值题容易召回错误行或错误单位 |
| 题目标签不统一 | `type` 中中英文混用 | 不能依赖标签硬编码 Prompt |
| 多文档关联 | 保险题最多 4 文档 | 需要证据聚合和冲突处理 |

## 9. 对系统设计的启示

第一，A 榜不需要先解决全量文档检索，但必须把证据检索、逐选项判断、Token 统计做扎实。A 榜可作为端到端闭环和错误分析集。

第二，B 榜重点是候选文档定位。应提前在全部 raw 文档上建立标题索引、正文 BM25、条款编号索引、数值索引和实体别名词典。

第三，直接全文上下文不可取。合同和财报单文档数十万字符，必须以检索和压缩为主；Qwen 1M 上下文可作为兜底能力，而不是默认策略。

第四，答案应由程序合成。模型只负责判断选项真假和给出证据，最终字母去重排序、单选截断、判断题合法化必须由后处理模块完成。

第五，证据输出不是附加项，而是调试核心。没有 `evidence.json` 很难定位错误来自解析、召回、压缩、推理还是后处理。

## 10. 建议的离线评估集

基于 A 榜构造内部评估维度：

| 维度 | 样本来源 | 指标 |
|---|---|---|
| 文档解析 | 每领域抽样文档 | 表格字段保真率、页码保留率、乱码率 |
| 检索召回 | 每题人工标注证据 | EvidenceRecall@5、EvidenceRecall@10 |
| 选项判断 | 由标准答案反推选项真假 | Option Accuracy、Option F1 |
| 答案格式 | 全部题 | InvalidAnswerRate |
| 成本 | 全部题 | AvgPromptTokens、TotalTokens |
| B榜模拟 | A题隐藏 doc_ids | DocRecall@5、DocRecall@10 |

## 11. 结论

数据分析表明，本赛题的本质是“长文档、多证据、选项级验证、严格格式化”的综合任务。A 榜题虽然提供 `doc_ids`，但文档本身非常长；B 榜将进一步放大文档定位难度。技术方案应避免依赖单一向量检索或长上下文暴力输入，而应采用结构化解析、多路检索、动态证据记忆、确定性计算和 Qwen 逐选项判断的组合策略。
